# Differences (`07f136a52e5bab02e45f90c1b26f849224d87864` vs `1f41fbc33caa4a982e7572a4b77ea3799a31cbff`)

- `apps/backend/src/services/sessionService.ts` in v1 owns both dialogue and generation; v2 splits that across `apps/backend/src/services/threads/threadConversationService.ts`, `threadGenerationService.ts`, `apps/backend/src/generation/orchestrator.ts`, and `apps/backend/src/pipeline/slotStages.ts`.
- v1 slot execution is one-shot and sequential: `apps/backend/src/generation/perSlotGenerator.ts::generateSingleProblem()` returns the whole draft, then `generateProblemsFromPlan()` retries the whole slot up to 3 times. v2 replaces that with staged retries in `apps/backend/src/pipeline/retryPolicy.ts` and never reruns the full slot as a unit.
- v1 fallback in `apps/backend/src/services/sessionService.ts::generateFromSession()` triggers on any terminal `GenerationSlotFailureError`; v2 fallback in `apps/backend/src/services/threads/threadGenerationService.ts::generateFromThread()` only runs when `allFailed` is true, so partial failures now persist as `INCOMPLETE` instead of being auto-salvaged.
- v1 repair can rewrite the whole generated draft and, for Python/C++/SQL, explicitly repair `test_suite` or `reference_solution` in `apps/backend/src/generation/perSlotGenerator.ts`; v2 post-Docker repair in `apps/backend/src/pipeline/slotStages.ts` only calls `generateReference(... role: "repair")` against a frozen test artifact.
- Most likely regression point: `apps/backend/src/pipeline/slotStages.ts` + `apps/backend/src/services/threads/threadGenerationService.ts` together remove two recovery paths that v1 had, so a single bad slot or a bad test/reference pairing is much more likely to terminate as partial failure instead of self-healing.
