# Handoff: 2026-04-01

## Summary

- Performed a deep infrastructure analysis of the current Ollama integration and one-button local fallback viability.
- Implemented the one-button Ollama fallback control plane across Electron main, preload, backend snapshot routing, and the renderer settings UI.

## Iteration Log

### Iteration 1 (Ollama One-Button Fallback Analysis)

- Traced the current LLM flow end to end:
  - renderer/preload/Electron main
  - engine IPC
  - session dialogue
  - generation pipeline
  - provider adapter dispatch
  (`apps/ide/main.js`, `apps/ide/preload.js`, `apps/backend/src/ipcServer.ts`, `apps/backend/src/services/sessionService.ts`, `apps/backend/src/services/dialogueService.ts`, `apps/backend/src/generation/index.ts`, `apps/backend/src/infra/llm/codemmProvider.ts`).
- Analyzed Ollama-specific lifecycle handling:
  - binary detection
  - server status check
  - best-effort `ollama serve`
  - `ollama pull`
  - install-link behavior
  (`apps/ide/main.js`, `apps/frontend/src/app/settings/llm/page.tsx`).
- Confirmed current fragmentation:
  - inference is abstracted behind the backend provider layer
  - install/start/pull/status live as Electron/UI-specific special cases
  (`apps/backend/src/infra/llm/adapters/ollama.ts`, `apps/backend/src/infra/llm/codemmProvider.ts`, `apps/ide/main.js`, `apps/frontend/src/app/settings/llm/page.tsx`).
- Verified that “Install Ollama” is currently only an external download link, not an automated install path.
- Identified readiness gaps, orchestration gaps, and state-coupling risks relevant to a one-button fallback flow.

### Iteration 2 (Production Architecture Design for One-Button Ollama Fallback)

- Designed a production-grade architecture for automated Ollama fallback with:
  - a central lifecycle orchestrator
  - an explicit runtime state machine
  - backend-visible readiness contracts
  - run-scoped provider snapshots
  - hardware-aware model selection
  - classified failure handling
- Scoped the design to current desktop constraints:
  - Electron main remains the host-operations owner
  - backend remains the engine / inference caller
  - renderer remains UI-only
  - engine API remains IPC-only.
- Documented the intended sequence from one user action to first successful local inference, including concurrency, cold-start, and degraded-runtime behavior.

### Iteration 3 (One-Button Ollama Fallback Implementation)

- Added a main-process local runtime control plane under `apps/ide/localLlm/`:
  - `LocalLlmOrchestrator`
  - `ollamaRuntimeDriver`
  - `hostCapabilityProbe`
  - `modelCatalog`
  - typed lifecycle errors
- Replaced the old provider-global runtime mutation path with run-scoped `llmSnapshot` injection:
  - Electron main resolves snapshots and leases
  - backend stores snapshots in `AsyncLocalStorage`
  - provider routing now pins model/provider per request
  (`apps/ide/main.js`, `apps/ide/preload.js`, `apps/backend/src/ipcServer.ts`, `apps/backend/src/infra/llm/executionContext.ts`, `apps/backend/src/infra/llm/codemmProvider.ts`).
- Removed app-path `CODEX_MODEL` overrides from dialogue, generation, repair, and edit flows so all features use the same resolved snapshot.
- Removed the old in-process runtime config module and `engine.configureLlm` handler.
- Replaced the settings UI with a one-button local activation flow that subscribes to orchestrator status instead of running lifecycle logic in the renderer.
- Added backend unit coverage for snapshot pinning and non-ready Ollama snapshot rejection.
- Updated architecture/docs to describe:
  - full local LLM architecture
  - lifecycle state machine
  - sequence flow
  - runtime flow
  - IPC contracts
  - snapshot + lease model
