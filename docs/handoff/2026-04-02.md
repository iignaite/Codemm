# Handoff: 2026-04-02

## Summary

- Investigated the new one-button Ollama integration after a reported runtime failure.
- Fixed a concrete macOS runtime-driver bug where Codemm looked for the wrong binary inside `Ollama.app`.

## Iteration Log

### Iteration 1 (macOS Ollama Binary Detection Fix)

- Audited the local runtime driver and found that macOS bundle detection targeted `Contents/MacOS/Ollama` instead of the Ollama CLI binary path used by the app bundle.
- Updated the runtime driver to prefer the CLI binary inside the macOS app bundle:
  - `/Applications/Ollama.app/Contents/Resources/ollama`
  - `~/Applications/Ollama.app/Contents/Resources/ollama`
- Kept `Contents/MacOS/Ollama` as a fallback candidate for compatibility with older/manual layouts.
- Updated the managed install path for the macOS download flow to point at `Contents/Resources/ollama`.

### Iteration 2 (Server Start Diagnostics + macOS Fallback Launch)

- Tightened `startServer()` in the Ollama runtime driver so Codemm no longer hides launch failures behind a generic timeout.
- Added captured launch diagnostics to `SERVER_START_FAILED` details:
  - attempted command
  - args
  - exit code / signal
  - stdout tail
  - stderr tail
- Added macOS-specific launch fallbacks when the binary lives inside an app bundle:
  - direct CLI `serve`
  - app executable `serve`
  - `open -a <Ollama.app>`
- Preserved timeout-based readiness waiting, but stopped treating a successful `open -a` exit as an immediate failure.

### Iteration 3 (macOS Managed Runtime Switch To CLI Binary)

- Changed the macOS managed-install flow away from downloading `Ollama.app` into Electron `userData`.
- The managed runtime now downloads the official standalone CLI archive instead:
  - `https://github.com/ollama/ollama/releases/latest/download/ollama-darwin.tgz`
- The archive is extracted into a managed `bin/ollama` path and marked executable before use.
- Existing system installs or app-bundle installs are still detected as fallbacks, but the managed runtime path is now a direct CLI binary rather than a GUI app bundle.
- This removes the macOS-specific failure mode where `open -a <managed Ollama.app>` succeeded but no Ollama server came up on `127.0.0.1:11434`.

### Iteration 4 (Already-Running Runtime State Transition Fix)

- Fixed a state-machine bug in `LocalLlmOrchestrator` when Ollama was already running before activation.
- Previously, the orchestrator could detect an installed + running runtime and jump directly from `INSTALLED` to `RUNNING`, which violated the declared transition graph and threw `INVALID_STATE_TRANSITION`.
- The existing-runtime path now transitions through `STARTING` with a "connecting to existing runtime" operation before entering `RUNNING`.

### Iteration 5 (Probe Warm-Up Timeout Hardening)

- Relaxed the Ollama readiness probe for cold-start model loads.
- `probeReadiness()` now retries on timeout with a longer warm-up window instead of failing immediately on the first 30-second timeout.
- Added structured probe failure details:
  - model
  - base URL
  - attempted timeout windows
- Reduced probe output size and set `keep_alive` so a successful warm-up leaves the model resident for follow-up calls.

### Iteration 6 (LLM Settings UX Simplification)

- Simplified the LLM settings page so the primary question is now obvious: what Codemm is currently using.
- The local card now emphasizes:
  - whether the local model is active
  - which model is being used
  - whether inference is currently routed locally
- Removed low-signal runtime details from the default view and moved them into a `Technical details` disclosure.
- Updated copy and button labels so local activation reads like a mode switch instead of a debugging panel.

### Iteration 7 (READY Chat Flow Guard + Benign IPC Fallback)

- Fixed the home chat flow after spec completion so the UI now treats `next_action: "ready"` as a terminal “spec is ready” response instead of falling back to another slot prompt.
- This removes the misleading experience where the user answered one extra follow-up after the backend had already finalized the activity spec.
- Hardened the Electron `codemm:threads:postMessage` bridge so if a late message arrives after the session is already `READY`, the bridge returns a benign ready payload instead of surfacing a hard renderer error.
- The end result is:
  - the chat explicitly tells the user to click `Generate`
  - duplicate late posts no longer show `Cannot post messages when session state is READY`
  - the home screen behaves consistently with the backend session state

### Iteration 8 (Generation Re-entry Lock + Retry Memory)

- Fixed a client-side generation race on the home screen where `Generate` could be invoked again before the React loading state had fully propagated.
- Added a synchronous generation ref guard so duplicate `generateV2` calls no longer race the backend into `Cannot generate when session state is GENERATING. Expected READY.`
- Strengthened the per-slot retry loop so repair prompts now carry a short failure history across attempts instead of only the latest failure.
- The repair context now remembers recent attempt failures with:
  - attempt number
  - phase (`contract`, `validate`, or `quality`)
  - message
  - repair strategy
  - obligation id when available
- Repair prompts now explicitly tell the model not to repeat the recent mistakes from prior attempts, which makes retries more stateful and less likely to thrash on the same invalid test-suite pattern.

## Known Issues / Next

- This fixes a concrete macOS pathing bug, but full runtime validation still requires testing on a machine with Ollama installed or with a real automated install run.
- If local activation still fails after this fix, the next step is to capture the exact failing stage from the local runtime status card (`INSTALLING`, `STARTING`, `PULLING_MODEL`, or `PROBING`) and tighten that part of the flow.

### Iteration 9 (Progress Stream Hint Fix)

- Fixed a misleading generation progress hint on the home screen.
- The UI previously showed `Progress stream unavailable.` after 1.2 seconds, which was too aggressive for cold-start local Ollama runs.
- Progress events were still arriving later, but the stale warning was never cleared, making the stream look broken even when it was working.
- The generation hint now waits longer before showing anything and uses softer copy: `Preparing generation... local models can take longer to emit the first update.`
- That hint is now cleared as soon as the first real progress event arrives.
