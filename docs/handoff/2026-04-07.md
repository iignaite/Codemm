# Handoff: 2026-04-07

## Summary

- Replaced the old exception-driven generation flow with a persistent run/slot state machine.
- Generation is now `runId`-scoped end to end and recovers deterministically after crashes.
- Judge execution now returns structured failure categories and uses stricter Docker sandboxing.

## Iteration Log

### Iteration 1 (Persistent Generation State Machine)

- Added persistent generation execution tables:
  - `generation_runs`
  - `generation_slot_runs`
  - `generation_slot_transitions`
- Added shared contracts for:
  - thread terminal generation states
  - generation run status
  - generation slot stages
  - generation slot terminal outcomes
- Refactored thread generation to:
  - create a durable run before slot execution
  - seed durable slot rows
  - persist slot-stage transitions from progress events
  - derive run/thread outcomes from slot results instead of the first thrown exception
- Added startup reconciliation so stale `PENDING` / `RUNNING` generation runs are repaired on engine boot and stuck thread state is rewritten.

### Iteration 2 (Run-Scoped IPC + Frontend Correlation)

- Reworked the generation progress bus to be keyed by `runId`.
- Updated backend thread IPC so:
  - `threads.generate`
  - `threads.generateV2`
  - `threads.subscribeGeneration`
  all accept or preserve `runId`.
- Updated Electron IPC/preload and frontend bridge clients so the UI subscribes to a specific generation run before it starts.
- Updated thread loading logic to surface latest persisted generation run metadata and accept the new terminal thread states.

### Iteration 3 (Judge Hardening + Failure Classification)

- Added structured judge failure categories:
  - `COMPILE_ERROR`
  - `TEST_FAILURE`
  - `EXEC_TIMEOUT`
  - `OUTPUT_LIMIT`
  - `INFRA_ERROR`
- Hardened Docker execution with:
  - explicit container names and forced cleanup
  - output limit detection
  - PID / CPU / memory limits
  - `no-new-privileges`
  - `cap-drop=ALL`
- Switched Java judging to compile into `/tmp/classes` so `/workspace` is mounted read-only.
- Added explicit compile-time and execution-time timeout markers inside judge commands so reference validation no longer collapses everything into a generic timeout.

### Iteration 4 (API Surface Cleanup, Tests, Docs)

- Deprecated fake targeted slot-regeneration strategies at the backend service boundary; only `retry_full_slot` is currently allowed until stage-targeted resume exists.
- Added unit coverage for:
  - run status aggregation
  - run-scoped progress isolation
  - timeout-stage judge classification
- Added integration coverage for stale-run reconciliation and stuck-thread recovery.
- Added `docs/architecture/GENERATION_STATE_MACHINE.md` and updated top-level docs to reflect the new model.

## How to Run

- Dev: `npm run dev`
- Full build: `npm run build`
- Backend unit tests for services: `npm --workspace codem-backend run test:unit -- services`
- Backend integration tests for DB flows: `npm --workspace codem-backend run test:integration -- database`

## Verification Run

- `npm run build`
- `npm --workspace codem-backend run test:unit -- services`
- `npm --workspace codem-backend run test:integration -- database`

## Known Issues / Next

- Stage-targeted slot resume is still not implemented; unsupported targeted strategy names are now rejected instead of silently rerunning full generation.
- The frontend still renders progress from events during active generation; longer term it should read persisted run/slot records directly for richer recovery and history views.
- Existing diagnostics still dual-write to legacy `runs` / `run_events`; that compatibility layer can be reduced once the new run tables back the UI directly.
